## Code health — dashboard (crash & slowness)

### 1. Crash causes (likely)

- **In-flight fetch after destroy (OpenTUI)**  
  The refresh `setInterval` callback does `if (renderer.isDestroyed) { clearInterval(timer); return; }` only at the **start** of the callback. It then `await fetchStatus(...)` (or `Promise.all([...])`). After destroy + `clearInterval`, that awaited promise can still resolve and run `result.match(refreshContent, ...)`. There is **no** `renderer.isDestroyed` check before calling `refreshContent` in the match callback. So a completion that lands after destroy can call `refreshContent` → `updateDefaultDashboardSections` / `replaceRootWithDashboardSections` / `replaceRootWithNewBox` and touch `renderer.root.getRenderable()`, `child.destroy()`, `renderer.root.add()`, `requestRender()`. That can throw or trigger undefined behaviour if the renderer/root is torn down.  
  **Locations:** `runOpenTUILive` (lines 586–609, 630), `runOpenTUILiveDashboardTasks` (734–775, 815), `runOpenTUILiveDashboardProjects` (918–962), and the other OpenTUI live functions that use the same pattern.

- **updateRootTextContent / updateDefaultDashboardSections on wrong shape**  
  Both assume a specific tree (root Box with one Text child, or three section boxes with content nodes). If OpenTUI ever replaces or reorders nodes (e.g. after an error box or a layout change), `getRenderable(rootId)`, `getChildren()`, or `children[0]` can be undefined or wrong type. Code does guard with `if (!box?.getChildren) return false` and similar, but then mutates `(textNode as { content: string }).content` or `node.content`. If the library has already replaced the node or the shape changed, that can throw. The outer `refreshContent` try/catch in `runOpenTUILive` swallows errors; other OpenTUI entry points also use try/catch in their refresh closures, so **uncaught** crashes from this are less likely but still possible if the throw is async (e.g. inside `requestRender?.()`).

- **Fallback: timer vs initial fetch**  
  In `runLiveFallbackDashboard`, `timer = setInterval(...)` is set immediately and a separate `fetchStatusData(config, ...).then(...)` runs. If the initial fetch fails, the error branch does `if (timer) clearInterval(timer)` and `process.exit(1)`. The interval is already running, so a later tick could still fire and call `write(...)` after the process has been told to exit — in practice usually mitigated by exit(1), but the ordering is racy.

- **Memory / listener leaks (long runs)**  
  - `createDiffWriter()` in `status.ts` registers `process.stdout.on("resize", ...)` and never removes it. Long-lived dashboard fallback keeps that listener and the ansi-diff instance alive.  
  - OpenTUI path: multiple `renderer.on("destroy", ...)` handlers (e.g. progressBar.stop + clearInterval, and the `resolve()` for the returned Promise). Not a leak per se, but cleanup is split and depends on OpenTUI emitting `destroy` once and consistently.

### 2. Slowness causes (likely)

- **Initial load**  
  - **OpenTUI:** `import("@opentui/core")` is raced with a **3s** timeout; `createCliRenderer()` with a **2s** timeout; then `renderer.setupTerminal()` with **SETUP_TIMEOUT_MS = 5s**. So worst-case ~3s + 2s + 5s before the first content fetch, and the first paint is delayed until the first fetch completes.  
  - **Loading progress:** `runLoadingProgressBar` uses a fixed **4s** duration (`DASHBOARD_LOADING_DURATION_MS`) and ticks every 80ms. The first real content is shown only when the initial fetch resolves and `progressBar.stop()` runs; if the fetch finishes in 1s, the user still sees the bar until that stop. So perceived load time is “max(actual fetch, progress bar behaviour)”.

- **fetchStatusData cost**  
  - Single `fetchStatusData` runs **many** parallel queries (globalCounts, agentMetrics, activePlans, actionablePerPlan, stale, plansAndActionable, statusCounts, next7, last7Completed, next7UpcomingPlans, last7CompletedPlans, activeWork, fetchStaleDoingTasks), plus conditional initiative/cycle.  
  - **execa path:** docs/infra.md and docs/performance.md state ~**150 ms per query** process-spawn overhead when not using the Dolt sql-server. With 13+ queries, that’s on the order of hundreds of ms to low seconds per refresh when the server isn’t used.  
  - **Cache:** `statusCacheTtlMs` default is **2.5s**; refresh interval is **2s**. So every 2s the cache may have expired and a full query set runs again. No backpressure: if a fetch takes 3s, the next tick still fires and starts another fetch.

- **Table rendering**  
  docs/cli-reference.md: *"Human-readable table output … can take 60–90+ seconds on large datasets"*. Dashboard uses the same formatting helpers (`formatStatusAsString`, `formatDashboardTasksView`, `formatDashboardProjectsView`, `getActivePlansSectionContent`, `getMergedActiveNextContent`, `renderTable`). For default dashboard, row counts are capped by `getDashboardRowLimitsDynamic` (e.g. `DASHBOARD_MAX_PLANS`, `DASHBOARD_MAX_TASKS`), so the **dashboard** view is partially protected, but the **--tasks** and **--projects** live views format full `activeTaskRows` / full status data without a hard cap in the formatting layer, so large datasets can make each refresh expensive.

- **Refresh path**  
  Every 2s the interval runs and (in OpenTUI) calls `fetchStatus` then `refreshContent` → `updateDefaultDashboardSections` or full replace. That does `getDefaultDashboardSectionContent`, `getActivePlansTableData`, `getMergedActiveNextTableData`, `buildTextTableContent`, etc. All of that is synchronous and runs on the main thread; with many plans/tasks it adds up. No “skip this tick if previous fetch not done” logic, so overlapping fetches and overlapping render work are possible.

### 3. Architecture / boundaries

- **Fallback vs OpenTUI**  
  - **Dashboard command** (`dashboard.ts`): if Bun, dynamically imports `./tui/live-opentui.js` and calls `runOpenTUILive` / `runOpenTUILiveDashboardTasks` / `runOpenTUILiveDashboardProjects`; on throw (e.g. “OpenTUI not available”) it falls back to `runLiveFallbackDashboard*`. Node always uses the fallback (no OpenTUI).  
  - Fallback: single shared pattern — stdin raw mode, SIGINT/SIGTERM + “q” cleanup, `createDiffWriter()` + `formatStatusAsString`/dashboard view helpers, `setInterval(..., REFRESH_MS)` with no backpressure.  
  - OpenTUI: 3s import timeout, 2s init timeout, 5s setupTerminal timeout; loading progress bar; single root (default dashboard) or single Box+Text (tasks/projects); refresh every 2s with `fetchStatus` then in-place update or replace; exit via SIGINT, “q” handler, and `renderer.on("destroy", () => resolve())`.

- **Cleanup on exit**  
  - Fallback: `cleanup()` clears the interval and restores stdin; registered on SIGINT and SIGTERM and on “q”/Ctrl+C.  
  - OpenTUI: SIGINT and prependInputHandler call `progressBar.stop()`, `renderer.destroy()`, `process.exit(0)`. The `destroy` handler clears the interval and resolves the Promise. As above, any in-flight fetch can still complete and call `refreshContent` after destroy.

- **Shared state / singletons**  
  - **getStatusCache()** in `status-cache.ts`: process-scoped singleton `_statusCache`. Dashboard and status both use it with `statusCacheTtlMs` (default 2.5s). Long-running dashboard reuses the same cache; no per-dashboard or per-session reset.  
  - **schemaFlagsCache** (Map by repoPath): 5-minute TTL; shared across all status/dashboard calls.  
  - Config: each refresh in the fallback uses `readConfig().asyncAndThen(...)` so config is re-read; OpenTUI path reuses the `config` passed in. No obvious staleness bug, but config is not re-read on the OpenTUI refresh path.

### 4. Suggested follow-up tasks (code health)

1. **Guard OpenTUI refresh completion on destroy**  
   In every OpenTUI live function, before calling `refreshContent` (or the error-path replace) inside `result.match(...)`, check `if (renderer.isDestroyed) return;`. Optionally also check at the top of `refreshContent` and in `updateRootTextContent` / `updateDefaultDashboardSections` so any late callback is a no-op once the renderer is destroyed.

2. **Add refresh backpressure**  
   Ensure only one fetch is in flight at a time: e.g. a “fetching” flag or abort/cancel of the previous request. If the 2s timer fires while a fetch is still running, skip starting a new one (or cancel the previous one and start a new one, with proper cleanup so no completion handler runs after destroy).

3. **Cap dashboard table data for live views**  
   For `tg dashboard --tasks` and `tg dashboard --projects`, cap the number of rows passed to `formatDashboardTasksView` / `formatDashboardProjectsView` (or equivalent) so that each refresh does bounded work (e.g. “Next 7” / “Last 7” already; cap “Active” to a maximum row count) and document the cap.

4. **Document server vs execa for dashboard load time**  
   In docs (e.g. cli-reference or a short “dashboard” section): recommend starting the Dolt sql-server (`tg server start` or equivalent) before using the dashboard to avoid ~150 ms/query execa overhead and to improve first load and every-2s refresh; mention that human-readable table output can be 60–90+ s on large datasets and that dashboard uses the same formatting with caps.

5. **Optional: remove or shorten loading progress when fetch is fast**  
   Consider stopping the loading progress bar as soon as the first fetch completes (already done) and avoid a minimum 4s bar when the fetch finishes in < 4s, so perceived initial load matches actual data readiness (e.g. stop on first content, not on a fixed duration).

## System health — task graph & dashboard ops

### 1. Task graph state

- **Known slowness** — Human-readable table output for `tg status --tasks` / `--projects` is documented to take **60–90+ seconds on large datasets** and can hit timeouts (`docs/cli-reference.md` § Performance / automation).
- **Recommendation** — For scripts and agents, use **`--json`**: `tg status --tasks --json`, `tg status --projects --json`, `tg next --json --limit N`. JSON skips table layout and returns much faster.
- **Recovery** — Stale `doing` or forgotten `todo`: use workflow Recovery (e.g. `tg done <taskId> --force --evidence "completed previously"`). Blocked tasks: `tg block <taskId> --on <blockerId> --reason "..."`.

### 2. Gate / lint / typecheck

- **Known gate:full failure** — **Three `status-live --json` tests** fail: `parseAsync` + `closeAllServerPools` + `process.exit(0)` in `src/cli/index.ts` can race with stdout flush when output is piped (`.cursor/memory.md`). Fix direction: use `process.exitCode = 0` and let the process exit naturally so stdout can drain; documented in `docs/testing.md` and `reports/26-03-02_review-*.md`.
- **test:all vs gate:full** — `pnpm test:all` runs `bun test __tests__ --concurrent` without db/mcp isolation; mock bleed can reappear. Either fix or avoid relying on test:all for “full” validation (memory.md).
- **Lint / typecheck** — `pnpm lint` (Biome), `pnpm typecheck` (changed files only), `pnpm typecheck:all` (full). Gate runs cheap-gate (lint + typecheck changed + affected tests). No other failure modes or flakiness called out in memory or docs for lint/typecheck.

### 3. Dependency and tooling (dashboard-relevant)

- **OpenTUI** — `@opentui/core` (package.json). Under Bun, the native Zig core can take **up to ~3s** on first load. Import timeout in `live-opentui.ts` is **3000 ms**, renderer init **2000 ms**. Reducing these (e.g. to 400 ms) causes silent fallback to the plain ansi-diff renderer (gray dashboard, no borders). Don’t lower these (memory.md, `docs/cli-tables.md`).
- **Bun** — Required (`engines.bun`). Dashboard TUI is tuned for Bun; Node falls back to setInterval + ANSI diff + boxen.
- **Dolt: server vs execa** — With **`TG_DOLT_SERVER_PORT` + `TG_DOLT_SERVER_DATABASE`** set: dashboard uses mysql2 pool (~5 ms/query). Without: execa path, **~150 ms/query**, serialized per repo to avoid noms lock. **Largest lever for dashboard speed**: run `pnpm tg server start` (or `dolt sql-server`) and set those env vars.
- **Caches** — **Status cache**: `TG_STATUS_CACHE_TTL_MS` default **2500 ms**; dashboard refresh is 2 s so repeat ticks often hit cache. **Query cache**: `TG_QUERY_CACHE_TTL_MS` default **0**; dashboard mode applies a **1500 ms** floor (infra.md). `TG_DISABLE_CACHE=1` disables status cache.

### 4. Operational readiness and suggested cleanup

- **Use `--json` for automation** — Any script, agent, or reprioritise flow should use `tg status --tasks --json`, `tg status --projects --json`, `tg next --json --limit N` so status/next are fast and avoid 60–90 s table rendering.
- **Run Dolt server for dashboard** — For a responsive dashboard, run `pnpm tg server start` (or equivalent) and set `TG_DOLT_SERVER_PORT` and `TG_DOLT_SERVER_DATABASE` so the CLI uses the pool instead of execa.
- **Dashboard Improvements plan** — `plans/26-03-03_dashboard_improvements.md` targets the “crashes after a while” and “slow to load” issues:
  - **Refresh backpressure** — Avoid overlapping `fetchStatusData` when the 2 s timer fires before the previous fetch finishes (guard in `dashboard.ts` and `live-opentui.ts`).
  - **Error visibility** — On fetch error, keep last good content and show error in a banner/footer instead of replacing the whole screen.
  - **Cache invalidation** — Add `getStatusCache().clear()` to all write commands (cancel, task, split, crossplan, gate, recover, plan, edge) so dashboard doesn’t serve stale data.
  - **Docs** — Document dashboard Dolt path (server vs execa), cache TTL, and refresh interval in `docs/performance.md`.
- **Gate:full** — Fix the CLI exit race in `src/cli/index.ts` (`process.exitCode = 0` + natural exit) so the three status-live --json tests pass and gate:full is reliably green.
- **Dolt recovery** — If you ever see “cannot resolve default branch head” or “active_branch() nil”, use the noms restore steps in `docs/infra.md` § Recovery; see `reports/dolt-default-branch-repair-2026-03-03.md` for repair behavior.
- **Stale doing / task hygiene** — If status shows stale doing tasks or you want a clean view, run Recovery steps from taskgraph-workflow (e.g. `tg done` with `--force` where work is already done), then re-check with `tg status --tasks --json` for speed.
