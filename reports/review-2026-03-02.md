### Architectural patterns

**Working:** Bottom-up data flow (CLI → domain/db/plan-import/export), single `AppError`/`ErrorCode`, Result types at boundaries, single `readConfig()` in cli/utils. Plan-import parser vs importer split is clear.

**Fuzzy:**

- **Domain vs “no DB”:** Rules say domain is “no DB”; in practice `domain/blocked-status.ts`, `domain/plan-completion.ts`, and `domain/invariants.ts` call db (escape/query). Either relax the rule or move DB-using logic into a small service layer.
- **Export → CLI:** `export/graph-data.ts` and `export/markdown.ts` import `Config`/`readConfig` from `../cli/utils`, so export depends on CLI. Prefer a shared `config` module used by both.
- **MCP → CLI:** MCP reuses cli/start, cli/status, cli/utils; acceptable but couples MCP to CLI structure.

### Risks and gaps

- **Orchestrator gate:full on main** — Must run gate:full from plan worktree; documented.
- **Plan branch before Wave 1** — Verify `plan-p-*` in `tg worktree list --json` after first `tg start --worktree`.
- **Hotspots:** `cli/status.ts` (2,281 lines), `db/migrate.ts` (1,071 lines) are single-file churn points.

## Three new significant improvements

These are the three highest-impact, actionable improvements suggested by the review.

### 1. Make gate:full reliably green (CLI exit race)

**What:** In `src/cli/index.ts`, stop using `process.exit(0)` on the success path after `parseAsync` and `closeAllServerPools()`. Set `process.exitCode = 0` and let the process exit naturally so stdout can flush when piped (e.g. in `status --tasks --json`).

**Why:** The three failing status-live --json integration tests are the known gate:full failure; this single change should make gate:full reliably green and restore CI/confidence.

**Follow-up:** Document in `docs/testing.md` or the agent field guide: “Do not use process.exit(0) on success when stdout may be piped.”

---

### 2. Extract config and fix export/cli layering

**What:** Introduce `src/config.ts` (or `src/shared/config.ts`) with `Config` and `readConfig()`. Have `cli/utils.ts` re-export or delegate so CLI call sites stay unchanged. Switch `export/graph-data.ts` and `export/markdown.ts` to import from config instead of `cli/utils`.

**Why:** Removes the “export depends on CLI” inversion, gives a single place for config (and future env overrides or multi-repo config), and makes export easier to test with a mock config.

---

### 3. Blocked-task triage pass

**What:** One-time triage of all blocked tasks: for each blocker, decide whether it’s still valid, can be canceled, or should become a small “unblock: …” task. Then unblock or cancel so at least one project has a clear runnable chain (e.g. Gate Full Triage or Breadcrumbs).

**Why:** Right now many tasks are blocked and none are “doing,” so the next /work wave is unclear. A triage pass makes the graph actionable and surfaces which projects are ready to grind.
